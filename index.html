Gobernanza y Riesgos

La gobernanza del proyecto open-source es un factor crítico para decisiones enterprise a largo plazo. Determina quién controla las actualizaciones, la seguridad, y la dirección del proyecto.

⚠️
Riesgo crítico: Gobernanza de WordPress

En septiembre 2024, Matt Mullenweg (CEO de Automattic y co-creador de WordPress) inició un conflicto público con WP Engine, exigiendo el 8% de sus ingresos por uso de la marca WordPress. Automattic bloqueó el acceso de WP Engine a recursos de WordPress.org, afectando actualizaciones de plugins y temas para millones de sitios. Esto expuso que una sola persona/entidad tiene control efectivo sobre el ecosistema WordPress.org, incluyendo el directorio de plugins, actualizaciones de seguridad, y la marca.

Fuentes: Brightspot — WordPress Dispute Impact · Ingenious — WordPress at a Crossroads

ℹ️
Contraste: Gobernanza de Drupal

Drupal es gobernado por la Drupal Association, una organización sin ánimo de lucro con gobernanza distribuida. Aunque Dries Buytaert (creador de Drupal y CTO de Acquia) tiene influencia significativa, la estructura de gobernanza es más transparente y no depende de una sola entidad comercial. El código, los módulos contribuidos, y la infraestructura del proyecto están separados de cualquier empresa individual.

Conclusión sobre el alcance

La PoC demostró gestión de textos/imágenes, preview en tiempo real e historial de versiones. Las demás capacidades necesarias para cubrir toda la compañía no fueron demostradas ni descartadas — es una PoC que podría escalarse, pero el esfuerzo y viabilidad requieren evaluación formal.

Riesgos a Evaluar

🔴 Bus Factor — A evaluar

Si el equipo clave rota, ¿quién mantiene el CMS? No hay comunidad externa. Mitigación: Documentación + pair programming + knowledge transfer.

🔴 Alcance transversal — A evaluar

La PoC cubre Tienda; viabilidad para Punto Com y otros activos no demostrada. Mitigación: POC extendido como siguiente paso.

🟠 Promoción entre ambientes — A evaluar

No hay mecanismo definido para DEV→STG→PRE→PROD. Mitigación: Definir antes de ir a producción.

🟠 Fragmentación — A evaluar

Si solo cubre Tienda, WordPress sigue para Punto Com = 2 sistemas. Mitigación: Exigir alcance transversal antes de aprobar.

Decisión Final — 11 junio 2026

Germán Sánchez (VP Tecnología) aprobó el desarrollo In-House del CMS. La decisión es definitiva. Este informe queda como registro histórico del proceso de evaluación y como base para el seguimiento del desarrollo.

Se desarrollará un CMS propio utilizando el stack de Tienda Digital (Java Spring Boot + Angular + MongoDB). El alcance inicial es Tienda Digital como piloto. La visión transversal fue confirmada ante las áreas funcionales el 12 de junio — .com, blog y todos los activos que usan WordPress migrarán una vez el piloto demuestre resultados.

  • Stack: Java Spring Boot + Angular + MongoDB (existente)
  • Equipo: Equipo de Ernesto Castillo (Tienda Digital)
  • Product Owner: Alejandro Llamas (PO actual de Tienda)
  • Repositorio: Nuevo repo dentro de cuentas de Tienda Digital
  • Observabilidad: Datadog (ya implementado)
  • Independencia de datos: Cada proyecto tiene datos independientes; comparten componente de administración
  • Demo a Germán: En 1 mes (julio 2026)
  • Transversalidad: Confirmada ante áreas funcionales (12 junio). Los resultados del piloto guiarán migraciones futuras.

Contexto Histórico: Lo que el análisis recomendaba

El análisis comparativo recomendaba Drupal Headless como opción con TCO verificable más bajo (USD 583,740 en 5 años) y capacidades enterprise demostradas. El desarrollo In-House se identificó como viable pero con riesgos controlables y capacidades no demostradas. La decisión final de Germán priorizó la alineación con el tech stack interno y la velocidad de ejecución del equipo de Tienda sobre las capacidades enterprise verificadas de Drupal.

Drupal sigue siendo la alternativa de respaldo si el desarrollo In-House no escala o no cubre las necesidades de otros dominios en el futuro.
